# 一种用于测试 extractors 的模式（草案）
针对 extractors 提取结果正确性的测试，从形式上来说，是一种集成测试。

在这种测试中，我们通常认为需要关注以下结果：
1. 在特定版本的高度段内是否正常
2. 在跨版本的高度段内是否正常
3. 在未来版本的高度段内是否正常

对于上述前两个目标，我们都可以通过建立特定的数据集来提供测试环境。
而对于第三个目标，需要有一种提前使新版本生效，或在预发布的环境中在线测序的方式。
我们接下来会针对不同情况分别描述。

## 数据集的构建
以一个覆盖 20 个高度的测试区间为例，我们需要提取这段高度内的：
- 所有 `BlockHeader`
- 所有 `Message`
- `StateTree` 上的所有数据

这种提取可以以类似 lotus 中的链数据导出的方式完成。一定程度上，我们可以将其视作某个 `snapshot` 的子集。
一旦数据集构建完成，我们可以任意地重复测试过程。因此，我们可以将基于数据集的测试视作静态测试。

## 未来版本的测试
未来版本的测试相对会复杂一点。正常情况下，我们可以通过参与公共的测试网络（如 calibnet）来进行。
这里提出一种离线测试的思路：
1. 确定好预备模拟未来版本的升级高度，在测试版本中，通过设置 lotus 中的 upgrade schedule 的方式来令升级提前生效
2. 准备好模拟升级高度前后一段高度内的 `BlockHeader` 和 `Message`，以及升级前的高度段内的 `StateTree` 全量数据。
3. 完成模拟高度段内的消息重放执行，令模拟升级高度之后的高度按照未来版本逻辑执行，并将数据固化下来。

这样我们就构建了一个“跨未来版本的高度段数据集”，之后的测试方式就和基于数据集的静态测试接近了。

## 测试的基本流程
1. 导入数据集，形成可用的 `ChainStore` 和 `StateManager`。
2. 基于 `ChainStore` 和 `StateManager` 进行指定高度范围内的提取
3. 通过人工或程序方式检视提取出的数据集，确认提取结果正确性

## 单元测试的方式
前面提到，基于数据集的测试是一种集成测试。
在此基础上，我们缩小数据集的体积，直到它仅包含一个特定 `State` 的必要数据时，这个数据集就可以视作一个`用例`，用在 extractor 级别的单元测试中。
考虑到单元测试需要覆盖“已出现的bug”，我们需要根据实际运行中出现过问题的场景，不断扩充用于单元测试的用例。


## 我们要做什么
- [ ] 一种基于链同步节点或链同步数据库提取指定高度范围内的测试数据集的方法，以及将数据集推送到 CI 可访问位置的自动流程
- [ ] 一种在中进行针对数据集的静态测试的方法，并与 CI 进行集成
- [ ] 一种基于指定的模拟升级高度和修改过的 upgrade schedule 产生模拟数据集的方法

这些内容或许将包括一个或多个命令行工具、一系列流程编排、CI 集成的工作，未来我们需要逐步完善这些内容。
